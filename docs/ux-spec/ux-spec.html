Design Direction Showcase

These directions explore how foundry-devil-code-sight could feel while staying faithful to the same UX foundation: native VS Code patterns, calm status visibility, unmistakable execution feedback, and a minimal Foundry companion surface.

Primary Goal Make write → run → verify feel faster and more trustworthy than Foundry's native macro editor.
Trust Contract Execution is only "ready" when connection is live and Foundry handshake is confirmed.
Visual Principle Use VS Code-native tone and semantic coloring rather than hard-coded branding.
Companion Surface Keep Foundry module UX minimal, native, and clearly secondary to the notebook loop.

VS Code Surface Map

VS Code owns the layout. Each surface below is fixed by VS Code conventions. The design decisions are only about the content within each surface — what to show, how to label it, and what level of detail to include.

Decision A: Status Bar — Two-Pill vs Single-Pill

The status bar is VS Code's surface. Our only decision is what text and semantic color appears inside our extension's status bar item(s). Connection and handshake can fail independently — they should be two distinct signals.

Status bar
VS Code — Status Bar Area (bottom of screen)
Option A1 — Two separate pills (recommended)
$(plug) Foundry: Ready $(check) Module OK
— both green: fully trusted, execute away
$(loading~spin) Foundry: Connecting $(circle-slash) Module: —
— connecting: module state unknown until handshake resolves
$(plug) Foundry: Ready $(warning) Module: Unsupported
— CDP alive but module needs update
$(plug) Foundry: Ready $(error) Module: Missing
— CDP alive but module not installed
$(debug-disconnect) Foundry: Disconnected $(circle-slash) Module: —
— no connection, both go inactive/grayed
Option A2 — Single combined pill
$(plug) Foundry Ready
— when everything is fine
$(warning) Module unsupported
— state collapses, less specific
$(debug-disconnect) Disconnected
— can't tell apart CDP vs module failure

Decision B: Notebook Cell Output — Format Variations

This is the most important design decision in V1. The cell output envelope is the primary feedback surface — every run must make success or failure unmistakable. Three approaches to consider.

Cell output
Notebook Cell Output — Format Variations
Option B1 — Minimal: value only (success) / error block (failure)
Success
{ name: "Goblin Scout", hp: 7 }
Error
TypeError: Cannot read properties of undefined
Line 2, Column 14
at eval:2:14
Cleanest. Matches raw Jupyter convention. Risk: success/failure not immediately distinguished by label.
Option B2 — Labelled: semantic header + value (approved)
✓ Result
{ name: "Goblin Scout", hp: 7 }
✕ Error · TypeError
Cannot read properties of undefined
Line 2 · Column 14
at eval:2:14
Semantic label + color signals state instantly. Error type surfaced at the header. Location detail on its own line. Satisfies both output clarity and error location requirements.
Option B3 — Structured envelope: all metadata explicit
✓ Execution Result
Status: Success · 0.31s
{ name: "Goblin Scout", hp: 7 }
✕ Execution Error
Type: TypeError · Line 2 · Column 14
Cannot read properties of undefined
Stack: at eval:2:14
Most information. Useful when debugging complex macros. Risk: adds label overhead for every simple run; may feel like overkill for the 80% case.

Decision C: Output Channel — $f.log() Entry Format

VS Code's Output Channel panel is a fixed surface. Our decision is what a $f.log() entry looks like inside it — enough to distinguish intentional output from noise without cluttering readable values.

Output channel
VS Code — Output Channel: Foundry Devil Code Sight
Option C1 — Timestamp + value
[12:04:31] Updated Goblin Scout HP to 5 [12:04:32] { combatActive: true, turn: 3 } [12:04:45] Rollback complete. Restored HP to 7
Option C2 — Prefix label + timestamp + value (approved)
[FOUNDRY-DCS 12:04:31] Updated Goblin Scout HP to 5 [FOUNDRY-DCS 12:04:32] { combatActive: true, turn: 3 } [FOUNDRY-DCS 12:04:45] Rollback complete. Restored HP to 7
Decision Recommended option Why
A: Status bar A2 — Single combined pill Compact status bar footprint with one authoritative readiness headline at a time.
B: Cell output B2 — Labelled semantic header Fast state recognition with clear success/error labeling and targeted diagnostics, without heavy metadata overhead.
C: Output channel C2 — Prefix label + timestamp + value Improved readability when scanning copied logs or mixed-channel workflows.
Step 9 approved direction: A2 (single-pill status bar) + B2 (labelled semantic cell output envelope) + C2 (prefixed timestamp log format). Companion module lives entirely inside Foundry's own settings UI; no custom VS Code side panel needed for V1.

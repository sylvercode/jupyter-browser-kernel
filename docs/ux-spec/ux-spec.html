Decision A: Status Signaling

Decision area evaluated for MVP: multi-indicator vs single authoritative status item.

Status Bar
VS Code status area mockup
Option A1 · Multi-indicator status items Alternative
$(plug) Connection: Connected $(check) Execution: Connected two independent signals when execution is viable
$(plug) Connection: Connected $(warning) Execution: Error good precision, heavier status-bar footprint
$(debug-disconnect) Connection: Disconnected $(circle-slash) Execution: Disconnected
More precise, but visually busier and less compact for routine use.
Option A2 · Single authoritative status item Approved
$(plug) Connected one authoritative session state headline
$(warning) Connecting concise progress or recoverable state
$(error) Error compact and readable at a glance
$(debug-disconnect) Disconnected inactive state when the session is detached
Single status reduces ambient noise while preserving readiness clarity.
Option A3 · Sentence-style status message Alternative
Connected. Connecting after reload. Error if recovery fails. Disconnected when the session detaches.
high clarity, weak scanability in a constrained status bar
Explicit wording, but too long for ambient status placement and truncates poorly.

Decision B: Cell Output Envelope

Decision area evaluated for MVP: minimal free-form output vs labeled semantic envelope.

Closest to raw notebook behavior, but success and failure are not immediately distinguished by label.
Option B2 · Labeled semantic envelope Approved
Result
{ ready: true, executionCount: 4 }
Error · TypeError
Cannot read properties of undefined Line 2 · Column 14 at eval:2:14
Balances clarity and speed with explicit labels and concise location-aware error detail.
Option B3 · Metadata-rich envelope Alternative
Execution Result
Status: Success · 0.31s Transport: CDP Value: { ready: true, executionCount: 4 }
Execution Error
Type: TypeError Line: 2 · Column: 14 Stack: at eval:2:14
Useful for deep debugging, but adds heavier metadata to routine iterations.

Decision C: Intentional Log Line Format

Decision area evaluated for MVP: unprefixed logs vs prefixed identity logs.

Output Channel
VS Code output channel mockup
Option C1 · Unprefixed log entries Alternative
[12:04:31] Execution started [12:04:32] { ready: true, step: 3 } [12:04:45] Reconnect complete
Cleanest line format, but weaker when copied or scanned outside the channel context.
Option C2 · Prefixed intentional log entries Approved
[JBK 12:04:31] Execution started [JBK 12:04:32] { ready: true, step: 3 } [JBK 12:04:45] Reconnect complete
Improves scanability and copy-search workflows without cluttering readable values.
Option C3 · Severity-rich prefixed logs Alternative
[INFO][runtime][12:04:31] Execution started [INFO][output][12:04:32] { ready: true, step: 3 } [INFO][session][12:04:45] Reconnect complete
More expressive, but heavier than needed for an MVP intentional output contract.
Decision Approved option Why
A: Status signaling A2 · Single authoritative status item Lowest ambient noise while preserving clear recovery states.
B: Cell output envelope B2 · Labeled semantic output envelope Balances clarity and speed with explicit labels and concise location-aware error detail.
C: Intentional log format C2 · Prefixed intentional log entries Improves scanability and copy-search workflows without cluttering readable values.
Section 8 approved direction: A2 for status signaling, B2 for notebook output, and C2 for intentional log entries. Source shards remain in docs/ux-spec, with canonical requirements in the PRD.
